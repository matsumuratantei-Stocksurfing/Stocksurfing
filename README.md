# 松村式Stocksurfing（クラウド版・Phase 3 / v3.6.0）

朝の仕掛け判定を自動化。GitHub Actions が毎朝7:30と寄り後9:20に指標を取得し、メール通知＋PWA表示する。

## URL（リポジトリ作成後に有効化）

`https://matsumuratantei-Stocksurfing.github.io/Stocksurfing/`

## アーキテクチャ

- `index.html` — メインアプリ（PWA対応）
- `fetch_data.py` — yfinance + Nikkei公式から12指標を取得
- `send_email.py` — Gmail SMTP で2宛先にメール送信
- `data.json` — fetch_data.py が出力。HTMLが読込
- `.github/workflows/fetch.yml` — 毎日2回スケジュール実行
- `manifest.webmanifest` + `sw.js` + `icons/` — PWA定義
- `requirements.txt` — Python依存

## スケジュール

| 時刻（JST） | 内容 |
|---|---|
| 7:30 | 場の判定スコア + 候補銘柄をメール |
| 9:20（平日） | 寄り後の最終判定をメール |
| 23:00（平日） | 答え合わせ自動実行（verify.yml） |
| 23:20（平日） | 観測モード 検出＋仮想決済（observe.yml） |
| 23:30（平日） | 一本化戦略 シグナル生成＋答え合わせ（unified.yml） |

## 奥様向け使い方

### 初回セットアップ（iPhone）

1. Safari で `https://matsumuratantei-Stocksurfing.github.io/Stocksurfing/` を開く
2. 画面下の **共有ボタン**（□に↑） をタップ
3. **「ホーム画面に追加」** をタップ
4. ホーム画面に「松村式Stocksurfing」アイコンが追加される

### 朝のワークフロー

1. **7:30頃** — メール「📈 [日付] 場の判定 +XX 追い風」が届く（プレビューで確認）
2. **8:30頃** — 必要があれば、ホーム画面のアイコンをタップしてアプリで詳細確認
3. **9:20頃** — メール「⏰ 寄り後判定 +XX」が届く
4. **9:25頃** — アプリで「✅ GO」「⚠️ 慎重に」「🛑 見送り推奨」のいずれかを確認
5. 仕掛けると判断したら、証券会社アプリで発注

### 売買ルール

- 損切：エントリー -2.0%（必ず逆指値設定）
- 利確①：+3% で半分利確
- 利確②：+5% で残り利確 or 引き上げトレール
- 同時保有：最大3銘柄
- 1銘柄あたり：総資金の20%以下
- 場が±20以内：様子見推奨

## トラブル時の対応

- **メールが来ない**：GitHub Actions の Run履歴を確認（リポジトリの「Actions」タブ）
- **アプリが古いデータ表示**：iPhone Safari でリロード（更新ボタン）
- **🔄自動取得 をタップしても変わらない**：data.json がまだ更新されていない可能性。次の定時実行を待つ

## v3.5.0 観測モード（前向き検証・発注なし）

「PO（パーフェクトオーダー）押し目エントリーに本当にエッジがあるか」を、**実際の発注を一切行わずに** 前向き（フォワード）検証する装置。3ヶ月以上のフォワード成績を貯め、事前固定の格上げ基準を全て満たしたときだけ、人間の判断でシグナル化を検討する。**これは機能追加ではなく検証実験の装置**であり、意図的に最小構成にしている。

### 3つのプロトコル原則

1. **前向き性（forward-looking）**：観測レコードは検出時点で入手可能なデータのみから生成する。未来のデータを参照するコードパスは存在しない（検出は `po_detector.detect_latest` で系列の最終日=当日までのみ判定）。
2. **不可変性（immutability）**：レコードは追記専用（append-only）。作成後の検出時フィールドの編集・削除機能もUIも持たない。決済時は close 項目を追記し status を更新するのみ。
3. **単一変数（single variable）**：検証する仮説は「PO押し目エントリーにエッジがあるか」だけ。損切りは Phase 1.5 で確定した構造損切り（swing_low×0.98）を観測期間中は変更せず流用する。

### 構成ファイル

- `po_detector.py` — PO検出＋構造損切りの**単一ソース**。バックテスト（`backtest_perfectorder.py`）と観測（`observe_*.py`）の両方がここから import する。発注・注文・証券APIは一切 import しない。
- `observe_detect.py` — 夜間ジョブ①。個別24＋セクターETF7の日足を取得し、その日にPO押し目が発火した銘柄を `observations.json` に**追記のみ**（間引きせず全件記録＝選択バイアス防止）。
- `observe_close.py` — 夜間ジョブ②。open レコードを構造損切りで仮想決済し、close 項目を追記。各レコードに dumb baseline（同銘柄・同保有日数の無条件売買）と TOPIX比 を併記。集計（`meta.summary`）と格上げ達成状況（`meta.promotion`）を再計算。
- `observe_criteria.py` — 格上げ基準 `PROMOTION_CRITERIA_V1`（開始前に確定・変更不可）と判定ロジック。
- `.github/workflows/observe.yml` — 平日23:20 JST（verify直後）に detect→close を実行し `observations.json` をコミット。
- アプリの `🔬観測` タブ — `observations.json` を読むだけの読み取り専用UI。「⚠️発注は行われません」バナー常時表示、直近検出・観測中オープン・クローズ成績・格上げ達成状況を表示。編集/削除/手動決済のUIは無い。

### 格上げ基準 PROMOTION_CRITERIA_V1（確定・開始後変更不可）

| # | 基準 | 閾値 |
|---|---|---|
| 1 | 観測期間 | ≥ 90 暦日 |
| 2 | クローズ済み件数 | ≥ 30 件 |
| 3 | 期待値（1回平均リターン） | > 0 |
| 4 | baseline比超過リターン | dumb / TOPIX の**両方** > 0 |
| 5 | R倍数（平均利益÷平均損失） | ≥ 1.5 |
| 6 | 弱い地合いサブグループの期待値 | 大幅マイナスでない（≥ -1.0%） |

全基準クリアで観測タブに「🎖 格上げ判定可能」バッジが出るが、**格上げ（シグナル化）は自動では行わない**。実行は必ず人間が相談して手動で決める。

### 🕒 時計リセット規約（logic_version reset）

観測期間中に検出ロジックまたは損切りロジックを変更した場合、`po_detector.py` の `LOGIC_VERSION` を必ず更新する（例：`po-v1` → `po-v2`）。集計（勝率・期待値・R倍数・baseline比・格上げ判定）は**現行 `LOGIC_VERSION` のクローズ済みレコードのみ**を対象とする。旧バージョンのレコードは保存されるが集計から除外される＝ロジックを変えたら実質「時計がリセット」され、90日・30件のカウントも新バージョンで貯め直しになる。これにより「途中でルールを変えて良い数字だけ拾う」後知恵バイアスを構造的に防ぐ。格上げ基準を変えたい場合も同様に `PROMOTION_CRITERIA_V2` を新設し、新基準は新バージョンの集計にのみ適用する。

## v3.6.0 一本化戦略（🎯タブ・検証中・発注なし）

「多くの要素を統合しても、結局ひとつの手法として機能する」ことを目指した松村式一本化戦略v1を、アプリが**銘柄選定→買い→売り→株数**まで一枚で出す `🎯一本化` タブとして実装。バックテストで勝率61%・1トレード期待値+3%を確認済み（`backtest_unified.py` / `unified_backtest_report.json`）。ただし上昇相場に偏った過去データのため、本番は前向き実測と併走させる「検証中」扱い。

### 一本化戦略v1（LOGIC_VERSION = unified-v1・確定）

5層を一本のルールに束ねたもの。各層は学術・実務で裏付けの厚い要素（順張りモメンタム／トレンドフィルター／低ボラ＝ボラ基準サイジング／トレンドフォロー出口）。

- **地合いフィルター**：TOPIX(1306)の終値 > TOPIX 200日線 の日だけエントリー可（下げ相場は全オフ）
- **入口（順張り）**：終値>200日線 かつ 200日線が上向き かつ（52週新高値 or 高値の3%以内）
- **損切り**：構造 = 直近10日安値 × 0.98（固定-2%は使わない）
- **利確/手仕舞い**：+5%で半分利確 → 残りは建値ストップ＆25日線割れでトレール（最長40営業日）
- **株数（サイジング）**：1トレードのリスク = 総資金 × RISK_PCT（松村さん確定=2%）を、(エントリー-損切り)幅で割って算出。荒い銘柄は少なく・穏やかな銘柄は多く＝損失額を一定に揃える。

### 3本のバックテストが出した最重要の発見

新高値・押し目・一本化のいずれの計測でも「**固定-2%の損切りが勝率を殺す**（勝率38%まで低下）。ボラ/構造に合わせた損切りにすると勝率61〜85%に跳ね上がる」。入口シグナルより**損切り設計とボラ基準サイジング**が最も再現性の高いエッジ、という結論。

### 構成ファイル

- `unified_signals.py` — 毎日ライブでv1の本日の買い候補を生成（`unified_signals.json`）。地合い・エントリー目安・構造損切り・+5%利確・25日線トレールを出力。発注なし。
- `unified_verify.py` — 答え合わせ。候補を初回だけ仮想トレードとして台帳(`unified_ledger.json`)に**append-only**で記録し、backtestと同じ出口で仮想決済→勝率/期待値/R/最大DDを集計＋格上げ基準 `UNIFIED_PROMOTION_V1`（90日・30件・勝率55%・期待値>0・R≥1.3）の達成状況を算出。**時計リセット規約**（LOGIC_VERSION変更で集計は新版のみ）を適用。
- `.github/workflows/unified.yml` — 平日23:30 JST（observe直後）に signals→verify を実行しコミット。
- アプリ `🎯一本化` タブ — 上記jsonを読むだけの読み取り専用。総資金＆リスク%を入れると候補ごとに株数を自動計算。検証成績・格上げ達成状況も表示。発注は一切しない。
- `backtest_unified.py` / `backtest_unified.yml` — 一本化戦略の計測（損切り種別・地合いON/OFF比較）。`backtest_newhigh.py` / `backtest_newhigh.yml` — 新高値エッジの計測。
